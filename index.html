The Evolution of Clean Architecture

Key Concepts & Principles that Informed Uncle Bob's Synthesis

1960s-1970s: Foundational Era
1968 :: Structured Programming
Edsger Dijkstra
"Go To Statement Considered Harmful" published in Communications of the ACM (March 1968). Advocated for clear, logical program structure using sequence, selection, and iteration instead of arbitrary jumps.
IMPACT ON CLEAN ARCHITECTURE:
Established the importance of clear, predictable code structures and control flow

1972 :: Information Hiding
David Parnas
"On the Criteria to Be Used in Decomposing Systems into Modules" (Communications of the ACM, December 1972). Modules should encapsulate design decisions and hide implementation details, making systems more maintainable.
IMPACT ON CLEAN ARCHITECTURE:
Direct ancestor to dependency rules and layer isolation principles

1980s-1990s: Object-Oriented Revolution
1987-1994 :: Liskov Substitution Principle
Barbara Liskov & Jeannette Wing
First presented in 1987 keynote "Data abstraction and hierarchy", formally articulated in 1994. Objects of a superclass should be replaceable with objects of its subclasses without breaking the application.
IMPACT ON CLEAN ARCHITECTURE:
Critical for plugin architecture and dependency inversion
ðŸ“„ A Behavioral Notion of Subtyping (ACM TOPLAS, 1994)
1992-2005 :: Hexagonal Architecture
Alistair Cockburn
Also called "Ports and Adapters". Developed in early 1990s, published online in 2005. Isolates core business logic from external concerns through ports (interfaces) and adapters (implementations), enabling test-driven development.
IMPACT ON CLEAN ARCHITECTURE:
MAJOR INFLUENCE: Concentric circles, dependency inversion, and isolation of business logic
ðŸ“„ Hexagonal Architecture (Alistair Cockburn, 2005)
1996 :: Dependency Inversion Principle
Robert C. Martin
Published in "The Dependency Inversion Principle" paper. High-level modules should not depend on low-level modules. Both should depend on abstractions. This principle became fundamental to modern software architecture.
IMPACT ON CLEAN ARCHITECTURE:
Foundation of the dependency rule - all dependencies point inward
ðŸ“„ The Dependency Inversion Principle (Object Mentor, 1996)
2000s: Architectural Patterns Mature
2000-2004 :: SOLID Principles
Robert C. Martin (coined by Michael Feathers)
Introduced in "Design Principles and Design Patterns" (2000). The acronym SOLID was coined by Michael Feathers around 2004. Five principles: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion.
IMPACT ON CLEAN ARCHITECTURE:
Tactical rules supporting Clean Architecture's strategic approach
ðŸ“„ Design Principles and Design Patterns (Object Mentor, 2000)
2003 :: Domain-Driven Design
Eric Evans
Published "Domain-Driven Design: Tackling Complexity in the Heart of Software". Emphasizes core domain logic, ubiquitous language, bounded contexts, entities, value objects, aggregates, and repositories.
IMPACT ON CLEAN ARCHITECTURE:
Entities layer design, use cases structure, and keeping domain logic at the center
ðŸ“š Domain-Driven Design: Tackling Complexity (Eric Evans, 2003)
2008 :: Onion Architecture
Jeffrey Palermo
First published in blog posts in 2008. Features concentric layers with domain model at the center, application services in middle layers, and infrastructure/UI on the outer layers. Emphasizes dependency inversion.
IMPACT ON CLEAN ARCHITECTURE:
Direct influence on the concentric circle diagram and inward-pointing dependencies
ðŸ“„ The Onion Architecture (Jeffrey Palermo, 2008)
2008-2012: Synthesis & Direct Precursors
2008-2011 :: Screaming Architecture
Robert C. Martin (Blog Posts)
Concept developed through multiple blog posts. Your architecture should scream the intent of the system, not the framework being used. The folder structure should reveal the domain, not the technical implementation.
IMPACT ON CLEAN ARCHITECTURE:
Business logic visibility over framework details
ðŸ“„ Screaming Architecture (Clean Coder Blog, 2011)
2012 :: The Clean Architecture
Robert C. Martin (Blog Post)
First articulation synthesizing Hexagonal, Onion, DCA, and BCE architectures into unified principles.
